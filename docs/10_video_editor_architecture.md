# ResearchReel — Video Editor & Rendering Studio Architecture

This document specifies the video editor module (`/editor/[draft_id]`), detailing the timeline, rendering engine, canvas composition, voiceover synthesis, and background transcoding pipelines.

## 2. Component Specifications

### 2.1 Timeline Tracks
* **Video Track**: Renders background elements (charts, animations, laboratory loops).
* **Audio Track**: Layered mixing for vocal narrations (TTS outputs) and background music tracks.
* **Subtitle Track**: Interactive timestamps matching subtitle files (WebVTT format) generated by text pacing.

### 2.2 Scene Script Editor (Sidebar)
* **Functionality**: Multi-scene editing grid. Allows users to write descriptions, modify speech text, upload slides/charts, and select voice profiles.

---

## 3. Technical Operations

### 3.1 Local State & Undo/Redo
* **Autosave**: The editor triggers autosaves to `PUT /api/v1/reels/draft/:id` every 30 seconds of inactivity.
* **Undo/Redo History**: Tracks operations in a local stack (limited to 50 operations).

### 3.2 Backend Rendering & Transcoding Pipeline
When the user clicks "Publish Reel":
1. **Scene Plan Serialization**: Frontend sends the timeline JSON to `POST /api/v1/reels/publish-draft/:id`.
2. **Audio Synthesis**: The Python worker checks if vocal segments are cached. Missing audios are generated via TTS.
3. **Asset Stitching**: FFmpeg runs image scaling, transition overlays, and audio merging:
   ```bash
   ffmpeg -loop 1 -t 10 -i scene1.png -i scene1_voice.mp3 -filter_complex "[0:v]scale=1080:1920[v];[1:a]volume=1.0[a]" -c:v libx264 -c:a aac output1.mp4
   ```
4. **HLS Conversion**: Combines scene MP4s and generates the multi-bitrate HLS playlist (`playlist.m3u8`).
5. **Notification**: The worker updates the database status and sends a Socket.IO event to the client: "Reel published successfully".

---

## 4. Storage & Performance
* **Storage Requirements**:
  * Uploaded images: Saved as compressed PNG/JPG (max 5MB).
  * Video outputs: 1080p HLS stream folder (average 15-20MB per 40-second reel).
* **Performance Considerations**:
  * Client side: Uses WebGL for canvas updates, preventing memory leaks during timeline scrubbing.
  * Server side: Relies on GPU workers (e.g. AWS G5) for FFmpeg processes. High-priority queues are reserved for Pro/Enterprise accounts.
